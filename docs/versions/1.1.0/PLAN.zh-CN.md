# ReadyType 1.1.0 计划

ReadyType 1.1.0 的目标是提升真实输入体验：更稳定地识别常用词，更自然地适配不同 App 的语气，并让触发方式和高精度语音包状态更容易理解。

需求边界见 [需求说明](./REQUIREMENTS.zh-CN.md)。当前代码已经有常用词、确认式建议、快捷键配置和高精度语音包状态的基础能力，本计划以产品化收口为主，不从零重写。

## 版本目标

- 让用户可以维护自己的常用词，减少人名、项目名、产品名和专业词识别错误。
- 在用户明确确认后，提供常用词学习建议，不做静默学习。
- 根据当前 App 和写作场景调整输出语气，优先解决聊天场景过于正式的问题。
- 支持自定义触发快捷键，默认仍保持双击 `Option`。
- 让高精度语音包的版本、状态和更新入口更清楚。
- 保持当前 DeepSeek-only、中文优先、低配置、低成本路线。

## 不做的事情

- 不新增多个大模型供应商。
- 不新增云端语音识别 API 配置。
- 不把模型选择暴露成工程师工具。
- 不做静默收集私人文本的自动学习。
- 不把 Liquid Glass HUD 作为 1.1.0 发布阻塞；只做技术方案和必要的小幅视觉改进。

## 功能拆分

### 1. 常用词库

用户价值：用户可以主动添加常用的人名、项目名、产品名、技术词和公司名。

产品要求：
- 设置中新增“常用词”区域。
- 支持添加、删除、查看常用词。
- 常用词按简单分类展示：人名、项目、产品、技术词、公司、其他。
- 文案使用“常用词”，不使用 hotwords、词典、训练等技术词。

技术要求：
- 复用现有 `UserVocabularyStore`、`ContextualVocabularyProvider`、`SmartTermDictionary`。
- 常用词保存到用户本地 Application Support，不写入仓库或日志。
- 识别和后处理都可以读取常用词，但不能明显增加首字延迟。

验收标准：
- 用户添加“ReadyType”和一个中文人名后，设置页能立即显示。
- 重启 App 后常用词仍存在。
- 删除常用词后不再出现在设置页。

### 2. 确认式学习建议

用户价值：当用户多次修正同一个词时，ReadyType 可以建议加入常用词，但必须由用户确认。

产品要求：
- 文案表达为“建议加入常用词”，不叫自学习、训练或机器学习。
- 不自动采集完整句子。
- 不把私人聊天、邮件正文作为学习内容保存。

技术要求：
- 复用 `UserVocabularyLearningService` 和 `UserVocabularySuggestionService`。
- 只保存候选短词、确认状态、分类和时间，不保存完整上下文。
- 增加去重、最短长度、最长长度和明显口头噪声过滤。

验收标准：
- 重复出现的候选词会进入待确认建议。
- 用户拒绝后不会立刻再次提示同一个词。
- 用户确认后进入常用词库。

### 3. App 场景语气

用户价值：在微信、邮件、笔记、文档和 AI 工具中，输出风格更符合使用场景。

产品要求：
- 聊天场景更自然，不默认加过度礼貌结尾。
- 邮件场景保留称呼、分段和落款建议。
- 笔记场景更适合条理化记录。
- AI 工具场景输出清晰任务说明。

技术要求：
- 复用 `OutputScenario`、`OutputContext`、`PromptTemplates`。
- 根据前台 App bundle id、窗口标题和用户手动场景选择推断场景。
- 场景推断不准时，以用户手动选择为准。

验收标准：
- 微信或聊天场景中，“谢谢”“麻烦你了”等结尾不被无根据添加。
- 通用、笔记和文档场景不无根据添加问候、致谢、确认、署名或收尾。
- 邮件意图能输出邮件格式。
- AI 工具场景能输出任务说明而不是闲聊文本。

### 4. 快捷键自定义

用户价值：避免和用户已有输入法、系统习惯或 App 快捷键冲突。

产品要求：
- 默认仍是双击 `Option`。
- 设置中提供清楚的“触发方式”配置。
- 修改后立即生效；失败时给出可理解提示。

技术要求：
- 复用 `VoiceShortcutConfiguration` 和 `GlobalShortcutService`。
- 配置保存在 `AppSettings`。
- 保留 `Esc` 取消行为。

验收标准：
- 默认双击 `Option` 可用。
- 用户切换触发方式后，新方式可用，旧方式不再触发。
- `Esc` 仍能取消当前语音输入。

### 5. 高精度语音包更新提示

用户价值：用户能清楚知道高精度语音包是否已安装、是否准备好、是否需要更新。

产品要求：
- 统一使用“高精度语音包”。
- 状态只显示真实状态，不写死“等待后台准备”。
- 提供删除、重新下载和版本说明。

技术要求：
- 扩展 `LocalSpeechModelManager`、`LocalSpeechModelReadiness`、`LocalSpeechModelDownloadService`。
- 记录本地包版本或清单版本。
- 更新检查作为独立状态，不覆盖“是否准备好”的真实状态。
- 当前默认使用内置清单检查；后续接远程清单时只替换清单获取器。
- 没有网络或无法检查更新时，显示“暂时无法检查”，不误报失败。

验收标准：
- 未安装、下载中、准备中、已准备好、检查失败、需要更新状态可区分。
- 准备好后不会回退成“尚未准备好”，除非文件确实缺失或损坏。
- 删除后状态立即变为未安装。

## 开发顺序

1. 常用词库 UI 和本地存储。
2. 确认式学习建议。
3. App 场景语气。
4. 快捷键自定义。
5. 高精度语音包更新提示。
6. 一轮真实 App 回归测试和文档更新。

## 测试计划

- `swift test`
- `scripts/build-app.sh`
- `scripts/package-dmg.sh`
- 手工测试微信、备忘录、浏览器、邮件/文档、AI 工具。
- 每轮真实语音测试记录：输入场景、原始识别、最终输出、耗时、是否自动粘贴。

## 发布标准

- 没有已知的 Esc 取消、双击触发、自动粘贴阻塞问题。
- 常用词和学习建议不保存完整私人文本。
- 高精度语音包状态与真实状态一致。
- README、Roadmap、Testing Guide、Troubleshooting 和 changelog 同步更新。
- Release 产物包含 DMG，并完成敏感信息检查。
