# ReadyType 1.2.0 实施计划：UI/UX Refresh

## 原则

1. 先冻结交互，再写 SwiftUI。
2. 先建立设计系统，再替换页面。
3. UI 重构不重写业务管线。
4. 每个阶段都保留可运行构建和回归门禁。
5. 高保真 UI 必须先逐页讲解并获得用户明确确认；用户回复“UI 确认，可以开发”之前，不修改应用 UI 代码。

## 阶段

### 1. 高保真原型

- 使用 Figma/Open Design 绘制首页、常用词、语言与输出、语音识别、权限与隐私、HUD 全状态。
- 同时提供浅色、深色和主题切换。
- 制作可点击主流程和 HUD 动效说明。
- 逐页讲解信息结构、交互、状态和取舍，根据反馈完成设计迭代并记录确认结论。

### 2. 设计系统

- 重构 `DesignSystem.swift`：语义颜色、材质、间距、字体和组件状态。
- 重构 `MotionTokens.swift`：HUD 状态时长、减少动态效果和过渡曲线。
- 新增外观偏好并持久化，默认跟随系统。

### 3. HUD

- 先重构 `RecordingHUDView` 和窗口尺寸策略。
- 用真实 App 验收出现、录音、识别、整理、输出、成功、复制、取消和错误。
- 保留 `Esc` 和粘贴流程不变。

### 4. 主窗口与菜单栏

- 将控制台改为首页摘要。
- 拆分设置页，避免单页滚动承载全部配置。
- 菜单栏只保留高频操作和状态。

### 5. App 感知

- 复用已有前台 App 和场景能力，新增用户可读短标签。
- 自动判断失败时使用通用整理，不改变现有输出安全约束。

### 6. 验收与发布

- Swift 单元测试覆盖主题偏好、状态展示和自动决策映射。
- Playwright/截图验收不适用于原生窗口时，使用现有 macOS UI 脚本和 Computer Use 截图。
- 覆盖浅色、深色、减少动态效果、不同分辨率和多 App 输入。
- 输出 1.2.0 中英文黑盒检查文档。

## 关键门禁

- `swift test`
- `scripts/build-app.sh`
- UI 文案与视觉截图门禁
- 微信、Mail/TextEdit、AI 工具真实输入
- `Esc`、自动粘贴、复制降级、高精度识别和 DeepSeek 回归

## 完成定义

只有当高保真原型已验收、三种外观可用、HUD 全状态通过、主窗口完成降噪、核心输入链路无回归时，才将 1.2.0 视为可发布。
